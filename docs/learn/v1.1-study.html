1. 一句话总览

v1.1 做了三件大事：统一规划链（/to-spec + /to-tickets 取代 to-prd / to-plan / to-issues 三个 skill）、wayfinder 转正（decision-mapping 更名重构后从 in-progress 毕业，定位为"超大工作量"的 situational on-ramp）、code-review 转正（带上 Fowler smell 基线进入主流程收尾）。周边还有 grilling / tdd / prototype 的行为收紧和新的 /research skill。

2. 改名对照表

旧名新名变化实质
/to-prd /to-spec 纯改名，"spec" 成为贯穿全套的统一术语（文档开头仍提 "you may know this as a PRD"）。产物文件也从 PRD.md 改为 spec.md。
/to-issues + /to-plan /to-tickets 两个 skill 合并为一个：把 spec 切成 tracer-bullet tickets，每张 ticket 声明自己的 blocking edges。同一份产物在 local tracker 是"一 ticket 一文件"，在真 tracker 是 native blocking links。
decision-mapping /wayfinder 更名 + 全面重构（fog of war / frontier / destination 的隐喻体系），从 in-progress 毕业进 engineering。
review（in-progress） /code-review 转正进 engineering，Standards 轴新增常驻的 Fowler smell 基线。

3. 最新主流程：idea → ship

idea→ /grill-with-docs→ /to-spec→ /to-tickets→ /implement×N→ ship
detour: /handoff→ /prototype→ /handoff （问题需要可运行的答案时，从 step 2 岔出去再回来）
/implement 内部: /tdd（red → green，逐 slice）→ /code-review→ commit
  1. /grill-with-docs — 有 codebase 时的起点：被拷问式访谈把想法磨尖，沉淀进 CONTEXT.md 和 ADR。没有 codebase 用 /grill-me（无状态版）。
  2. 分支：问题能全靠对话敲定吗？ 需要"跑起来才知道"的答案（状态模型、业务逻辑、UI 长相）→ 用 /handoff 岔出去开 /prototype 新会话，答案再 handoff 回来。
  3. 分支：是多 session 的活吗？
    • 是 → /to-spec 把对话综合成 spec（不再访谈，只综合），然后 /to-tickets 切成带 blocking edges 的 tickets → 每张 ticket 单独开 fresh context 跑 /implement。
    • 否 → 同一个 context window 里直接 /implement。
  4. /implement 内部在事先约定的 seam 上驱动 /tdd（一次一个 red-green slice），收尾跑 /code-review 再 commit。

三个 on-ramp（从别的起点汇入主流程）

/triage

bug 报告、外部需求堆积时用。按角色状态机把 raw issue 变成 agent-ready issue。只 triage 你没创建的 issue —— /to-tickets 产出的 tickets 天生 agent-ready，别再过 triage。v1.1 起也能 triage 外部 PR。

/diagnosing-bugs

东西坏了、难查的 bug、偶发 flake。先建立紧反馈环（一条命令能在这个 bug 上变红）才允许推理，修复必带回归测试。post-mortem 发现"没有好 seam 锁住 bug"时交棒 /improve-codebase-architecture。

/wayfinder

超大且迷雾重重的工程 —— greenfield 项目、一个 session 装不下的大 feature。在 issue tracker 上铺一张调查 tickets 的共享地图，一次解决一张，直到迷雾散尽，再汇入 /to-spec。

4. 逐项变更详解

规划链统一：/to-spec + /to-ticketsminor

/to-spec：把当前对话综合成 spec 并发布到 issue tracker。三步：探索 repo（用 domain glossary 词汇、尊重 ADR）→ 先和你确认测试 seam（优先复用既有 seam，越少越高越好，理想是 1 个）→ 按模板写 spec（Problem / Solution / 大量 User Stories / Implementation Decisions / Testing Decisions / Out of Scope），打 ready-for-agent 标签。

/to-tickets：把 spec / 对话切成 tracer-bullet 垂直切片，每片纵穿所有层（schema、API、UI、tests），单独可 demo，尺寸恰好一个 fresh context window。每张 ticket 声明 blocking edges，先给你过目（粒度对吗？依赖边对吗？）再发布：

  • local tracker → 一 ticket 一文件：.scratch/<feature>/issues/<NN>-<slug>.md，依赖顺序编号，手工自上而下做。
  • 真 tracker（GitHub / Linear…）→ 一 ticket 一 issue，优先用平台原生的 sub-issue / blocking 关系，于是"frontier（所有 blocker 已完成的 ticket）"可以多个 agent 并行抢。

Wide refactor 例外：一个机械改动（改列名、改共享类型）blast radius 波及全库时，垂直切片切不绿 —— 改用 expand–contract：先 expand（新旧并存）→ 按 blast radius 分批 migrate（每批一张 ticket，都被 expand block）→ 最后 contract（删旧形态，被所有 migrate 批 block），CI 逐批保绿。

/wayfinder 毕业 + 重构minor

核心隐喻：向 destination（目的地）寻路。它规划、不动手 —— 每张 ticket 解决一个决策而非交付物，地图完成的标志是"动手之前没有任何事待决定"。

  • 地图是索引不是仓库：一个 wayfinder:map issue，tickets 是它的 child issues，全队共享一个 URL。决策只存在于它自己的 ticket 里，地图只做摘要和链接。
  • Fog of war：能精确说出问题（不必能回答）→ 建 ticket；说不清 → 留在 ## Not yet specified 迷雾区，随前沿推进逐步"毕业"成新 ticket。超出目的地的工作是 ## Out of scope，永不毕业。
  • 四种 ticket 类型：grilling（默认，一次一问）、prototype、research、task（唯一"做事"的类型，只为解锁某个决策）。
  • HITL / AFK 分类：grilling、prototype 是 HITL（human in the loop），只能通过和人的实时交流 resolve —— 这修掉了"wayfinder 自己拷问自己"的老毛病；research 是 AFK。
  • 认领即 assign：session 干活前先把 ticket assign 给驱动的 dev，防止并行 session 相撞。铁律：一个 session 最多 resolve 一张 ticket。
  • 无雾早退：开场 grilling 没发现迷雾 → 直接停，告诉你这活一个 session 就够，不建没人需要的地图。
  • 定位：situational on-ramp，不是新的主入口 —— grill 主导的 idea → ship 仍是正门。

/code-review 转正 + Fowler smell 基线minor

两轴并行 sub-agent review：Standards（符合本 repo 成文规范吗？）+ Spec（忠实实现了源头 issue / spec 吗？）。Standards 轴新增常驻基线：约 12 个高信号的 Fowler "Bad Smells in Code"（Mysterious Name、Duplicated Code、Feature Envy、Data Clumps、Primitive Obsession、Repeated Switches、Shotgun Surgery、Divergent Change、Speculative Generality、Message Chains、Middle Man、Refused Bequest）。两条安全规则：repo 成文规范覆盖基线；smell 只作为 judgement call 报告，从不算硬违规。

/grilling 收紧minor

  • 确认门：你确认"达成共同理解"之前，agent 不许动手执行计划。
  • facts vs decisions：事实自己查（探索 codebase），决策必须交给人并等待回答。防止 grilling 被别的 skill 内嵌调用时抢答自己的问题。

/tdd 瘦身为 reference-onlypatch

  • 删掉逐步 Workflow —— red → green 循环模型本来就会。refactor 阶段被移除：重构归属 review 阶段（/code-review）。
  • seam 成为测试落点的关键词：只在事先与你确认过的 seam 上写测试。
  • 新反模式：tautological test —— 断言用和实现同样的方式重新算出来，构造上必过、零信心。期望值必须来自独立的 source of truth。

/prototype：model-invoked + primary sourceminor

agent 现在可以自主够到它（其他 skill 也可以）。新观念：prototype 是 primary source —— 回答完设计问题后不再删掉，而是作为可运行证据留在 main 之外的 prototype/<name> 分支上，实现 issue 上留 context 指针。main 只保留验证过的决策，探索过程可回溯。

/research新 skill

把"读资料的腿脚活"外包给后台 agent：它对照 primary sources（官方文档、源码、spec、第一方 API）调查一个问题，在 repo 里留下一份带引用的 Markdown。你继续干活，回头拿文档来 grill / 规划 / 设计 —— research 喂养思考，不替代思考。

其余更新

  • /ask-matt 路由补全：新增 "Something's broken" on-ramp（diagnosing-bugs）、"Vocabulary underneath" 区（domain-modeling + codebase-design），tdd 织入主流程。迷路时问它就对了。
  • /triage 支持外部 PR：PR 被当作"带代码的 issue"走同一套角色状态机（按 repo 开关启用）；已实现的请求直接 resolve 为 wontfix。
  • /setup-matt-pocock-skills 更友好：triage labels 只在装了 triage 时才问且默认推荐 yes；domain docs 默认单 context（除非 monorepo 信号）;local tickets 与 to-tickets 对齐为一文件一 ticket。
  • /writing-great-skills 新增两个 failure mode：Negation（大象效应 —— 说"别做 X"反而把 X 拽进上下文，要写正向指令）与 Negative Space（留白盲区 —— 每个没写的决定都被默默委托给模型先验，要主动审视草稿的沉默处）。
  • setup-ts-deep-modulesin-progress：给 TypeScript repo 装 dependency-cruiser，强制每个 package 成为 deep module（实现藏在子目录，只能从 entry-point 文件进入）。草稿阶段，未打包。

6. Context 卫生守则

这套 skill 的隐藏主线是 context window 管理 —— 每个环节都在回答"这段思考应该活在哪个 window 里"。
  • grilling → to-spec → to-tickets 保持一个不间断的 context window，中途别 compact 别 clear —— 拷问、spec、切票要建立在同一段思考上。
  • 上限是 smart zone（约 120k tokens，模型仍能敏锐推理的窗口）。切票前就逼近了？别硬撑降智 —— /handoff 换新线程继续。
  • 每张 ticket 的 /implement 都从 fresh context 开始，做完一张清一次上下文。
  • /handoff 是分叉（压缩成 markdown 文件，新会话引用它）；/compact 是原地续命（只在阶段间歇用，别在阶段中途 compact）。
  • wayfinder 铁律同源：一个 session 只 resolve 一张 ticket —— session 就是 context window。
